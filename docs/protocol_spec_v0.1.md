GNAF (Guild Network Application Framework)

Protocol Specification v0.1



1\. System Overview

GNAF is a hierarchical, federated labor execution network that combines:

\- contract-based work execution

\- reputation-weighted skill progression

\- guild-based governance structures

\- jurisdictional fallback arbitration (country layer)



The system prioritizes decentralized execution with centralized rule inheritance and minimal sovereign intervention.



2\. Core Entities



2.1 Guild

A Guild is a governance and execution container that defines:

\- rule set (inherited from parent guild if applicable)

\- membership structure

\- contract participation eligibility rules

\- internal ranking system



Guild ownership defines:

\- rule enforcement authority

\- activation approval of sub-guilds



2.2 Sub-Guild

A Sub-Guild is a delegated execution unit that:

\- inherits all parent guild rules

\- operates under guild owner approval

\- contains members within -3 rank threshold of host



2.3 Member

A Member is an individual node in the system with:

\- rank within a guild

\- job completion history (guild-specific experience)

\- eligibility for contracts and sub-guild participation



2.4 Contract

A Contract is a reputation-aware escrow execution agreement that includes:

\- locked funds held within GNAF

\- contractor and contractee participation

\- validation-based completion logic

\- optional weighted experience assignment



2.5 Guild Experience Unit

Each completed job contributes:

\- +1 base experience within guild context

\- optional weighted adjustment (requires contractor agreement)



Experience is guild-local, not global.



2.6 Contractor (Optional Role Extension)

A Contractor may opt into:

\- reputation tracking

\- job difficulty estimation tools

\- visibility of failure patterns across rank bands



2.7 Country / Jurisdiction Layer

A sovereign oversight layer that activates only under:

\- guild ownership disputes

\- guild inactivity conditions



Responsible for ownership succession protocol.



3\. Governance Model



3.1 Rule Inheritance

\- Sub-guilds inherit all rules from parent guild

\- Rules cannot be independently modified by sub-guilds

\- Rule consistency enforced across hierarchy



3.2 Guild Activation Control

\- Sub-guilds require guild owner approval before execution



3.3 Ownership Succession Protocol

Triggered only under jurisdiction layer:

1\. Detect dispute/inactivity

2\. Identify candidate pool

3\. Competition for ownership

4\. Winner becomes guild owner



4\. Reputation \& Experience System



4.1 Guild Experience Tracking

\- Each job completion = +1 experience unit

\- Guild-local progression system



4.2 Weighted Experience

\- Optional weighting per job

\- Requires external agreement (typically contractor)



4.3 Failure Visibility System

\- Visible only within ±2 rank band



4.4 Failure-Based System Behavior

\- Contractor may request refund

\- Contract flagged and optionally rerouted

\- If refunded: contract removed from platform



5\. Guild Hierarchy



5.1 Rank System

\- Rank determines access and eligibility



5.2 Sub-Guild Leadership Rule

\- Members may form sub-guilds if within rank conditions

\- Sub-guild members: -3 rank threshold



6\. Transparency Model



6.1 Visibility Constraints

\- Failure data visible only within ±2 rank band



6.2 Contractor Visibility

\- Optional aggregated analytics for contractors



7\. System Behavior Summary

GNAF is a recursive labor execution network with:

\- escrowed contracts

\- guild-local reputation systems

\- controlled transparency

\- sovereign fallback governance



8\. Core Principles

\- Local truth over global abstraction

\- Rule inheritance over divergence

\- Failure as learning data

\- Decentralized execution

\- Centralized governance inheritance

\- Sovereign intervention only on failure
