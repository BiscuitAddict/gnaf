GNAF — CONTRACT EXECUTION ENGINE

1. TASK LIFECYCLE

Created → Requested → On the Board → Submitted → Verified → Hold Period → Funds Released

States:

- CREATED
  Contract is defined by origin guild

- REQUESTED
  Sent to executing guild

- ON THE BOARD
  Placed in guild internal task board for member assignment

- SUBMITTED
  Work completed and delivered

- VERIFIED
  Contractor confirms or rejects output

- HOLD (24–48 HOURS)
  Anti-fraud / anti-reneging protection window

- FUNDS RELEASED
  GNA-C settlement finalized


2. INTERNAL GUILD ASSIGNMENT

- Work is posted to a Guild Task Board
- Members self-select based on skill level + eligibility
- Guild leadership may override assignment
- Assignment is internal only (no external workers)


3. WORK PHILOSOPHY

- All tasks are educational-first
- Payment is secondary and optional incentive
- Primary value = skill development + documented experience
- Contractors validate quality, not just completion


4. PROOF OF WORK STANDARD

Each submission must include:

A. EXECUTION REPORT (Scientific Format)
- Objective of task
- Method used
- Steps taken
- Results achieved
- Errors or constraints encountered
- Conclusion / outcome summary

B. SUPPORTING ARTIFACTS (if applicable)
- Logs
- Screenshots
- Data outputs
- System traces
- Calculations or measurements


5. VERIFICATION SYSTEM

- Contractor reviews submission
- Either:
  • APPROVED
  • REJECTED (with reason tag, not narrative required)

- Optional guild internal review layer allowed


6. HOLD PERIOD RULE

- After verification, funds are locked for 24–48 hours
- Purpose:
  • dispute prevention
  • fraud correction window
  • re-evaluation buffer

- After hold expires:
  → settlement becomes final


7. SYSTEM OUTCOME PRINCIPLE

- Every contract becomes a documented learning artifact
- Guilds accumulate operational knowledge over time
- Work quality improves through repeated structured execution
- Payment acts as reinforcement, not primary motivation
